# 9. decay model

## status

accepted

date: 2026-05-30

## context

v0.2 introduces scheduled context maintenance: runs that have passed their ttl get processed
according to their confidence tier. the options are:

- a separate `compost` table (physically separates decayed records from live ones)
- in-place state on the `runs` table (a `decay_state` column marks what happened)

the choice affects query patterns, projection generation, and idempotency requirements.

## decision

decay is applied **in-place** on `runs` via a `decay_state` column, keeping the single-table
canonical-store pattern. there is no separate compost table.

four orthogonal behaviors based on confidence tier:

| confidence | behavior after ttl | decay_state | projection |
| --- | --- | --- | --- |
| `solid` | retained as-is | `'retained'` | stays in SUBSTRATE.md |
| `directional` | body compressed into `distilled_summary`, original summary dropped | `'distilled'` | moves to COMPOST.md |
| `vibes` | body dropped | `'decayed'` | moves to COMPOST.md |
| any + `no_compost = 1` | preserved 100%, untouched | NULL (stays live) | stays in SUBSTRATE.md |

the `no_compost` flag is **orthogonal** to confidence. when set, maintenance skips the record
entirely regardless of ttl or tier.

maintenance only acts on rows where:

```
decay_state IS NULL
AND expires_at <= now
AND no_compost = 0
```

this makes maintenance **idempotent**: re-running it on already-processed rows is a no-op.

## rationale

in-place decay keeps the query surface simple: all runs live in one table, filterable by
`decay_state`. a separate compost table would require joins for cross-state queries and would
complicate projection regeneration from scratch.

idempotency follows naturally: the filter `decay_state IS NULL` means processed rows are never
re-evaluated.

orthogonal `no_compost` keeps preservation behavior explicit and inspectable without needing a
separate exclusion list.

## consequences

- all runs, live and decayed, are queryable in one place. **confidence: solid. load-bearing.**
- COMPOST.md projection is generated from `decay_state IN ('distilled', 'decayed')` rows.
- SUBSTRATE.md projection covers `decay_state IS NULL OR decay_state = 'retained'` rows.
- `distilled_summary` must be populated before `decay_state` is set to `'distilled'`. ordering
  matters; the maintenance engine is responsible for the transaction. **confidence: solid.**
- hard-deleting a decayed row is always valid later without breaking the model.
- the schema can add new `decay_state` values in future versions without breaking existing queries.

## unresolved

- whether `distilled_summary` is generated by an LLM call or a deterministic rule in v0.2.
- exact COMPOST.md projection format.
- whether maintenance runs in a transaction per-record or batched.
